# 反 AI 味清单（design-brief-builder）

> 定稿前读本文。不是为了"检查"，是为了**激活正确的设计表达模式**。
> 下面三段锚点覆盖了 design-brief 最常见的三种场景。

---

## 锚点一：设计方向描述模式

> 模型的自然倾向是堆砌形容词（"现代感""高级""简约"）。
> 下面展示正确的设计表达：抽象感受 → 具体设计属性。

```markdown
## ❌ 模糊描述
"整体风格简约现代，有高级感。"

## ✅ 具体描述
**关键词**: 干净、高效、数据密集

**颜色**: 冷色调 + 白色背景，品牌色 #2563EB 只在可交互元素上使用
**排版**: Inter 字体，标题 20px/600，正文 14px/400，辅助 12px/400
**间距**: 8px 网格系统，卡片间距 24px，内边距 16px
**圆角**: 按钮 8px，卡片 12px，模态框 16px
**信息密度**: 高——页面同时展示 30+ 行数据，不追求大面积留白
```

关键点：
- "简约现代" 不是设计方向——颜色、字体、间距、圆角的具体数值才是
- 每个抽象形容词都要翻译成具体的设计 token

---

## 锚点二：参考资料分析模式

> 模型的自然倾向是只写参考产品名，不分析借鉴了哪一点。
> 下面展示正确的 reference 分析。

```markdown
## ❌ 只说名字
"参考 Linear 的设计风格"

## ✅ 分析具体
| Product | What we like | Why |
|---------|-------------|-----|
| Linear | 侧边栏变体导航 + 内容区无分割线全宽布局 | 50+ 页面结构适用 |
| Notion | 空状态引导设计（"添加你的第一个项目"） | 降低新用户上手成本 |
| Vercel | 暗色模式下的颜色对比度管理 | 我们的用户也大量使用暗色 |
```

关键点：
- 只说产品名 = 把设计决策推给模型脑补
- 写明具体借鉴了什么 + 为什么适合本项目

---

## 锚点三：状态设计兜底模式

> 模型的自然倾向是只设计默认状态（happy path），忽略异常状态。
> 下面展示正确的状态覆盖检查。

```markdown
## State Design Direction

| 状态 | 方向 |
|------|------|
| 默认 | 数据正常展示 |
| 空 | "尚无数据" + 引导创建（参考 Notion 风格）|
| 加载 | skeleton（非 spinner）|
| 错误 | 具体错误信息 + 重试按钮 |
| 权限不足 | 403 页面，不暴露敏感信息 |
```

关键点：
- 空/加载/错误状态不能 [TBD]
- 每个状态的设计方向必须有具体描述

---

## 兜底检查

| 检查项 | 通过标准 |
|--------|----------|
| 渐变滥用 | 无"默认紫蓝对角渐变 Hero"作为唯一视觉手段 |
| 玻璃拟态堆叠 | 非多层半透明卡片套娃且无功能理由 |
| 假 3D / 无意义插画 | 插画服务信息或品牌，非装饰填充 |
| 密度与功能一致 | 信息密度高时不强行大面积留白——"极简"不是所有场景的答案 |
| 字体层级 | 标题/正文/辅助至少 3 级 |
| 对标具体 | Reference 写清喜欢哪一点，非只写产品名 |
| 可执行性 | 开发能据此选 Tailwind/组件风格，非仅形容词 |

**Brief 末尾**：增加 `## Anti-Slop Review` — 列出已排除的 slop 模式（1–3 条即可）。
