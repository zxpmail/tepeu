# First Principles — Domain Mapper

> 不是行业研究工具。是**非结构化信息 → 结构化 Markdown 数据库**的通用管线。
>
> 延伸阅读：领域快照之上的跨文档模式识别 → [core/docs/bottleneck-pattern.md](../../../docs/bottleneck-pattern.md)（HBM / 公众号 / ReqForge 三个案例共享的瓶颈结构）

## 核心理念

```
Scattered info（官网 / Reddit / 财报 / 社媒 / 日志 / 代码库）
  → 结构化 Markdown DB（domain-map.md）
```

所有领域的"信息压结构"遵循同一管线，区别只是输入源和输出模板字段不同。

## 锚点案例：HBM 高带宽内存

> 这个技能之所以存在，是因为下面这个案例证明了"信息管线"的通吃能力。

### 问题
让 Sonnet/Opus 研究一个陌生行业，往往会得到一篇漂亮但空洞的散文——看起来说了很多，细看没有一句能用的。

### 解法（3 步代替 1 步）
不是"帮我研究 X，输出报告"——而是分三步走，每一步的输出是一份结构化文件：

**Step 1: 行业数据库**
```
输入："帮我研究 HBM 行业"
输出：hbm-industry.md
  ├─ 主要品牌
  ├─ 产品分代（HBM1 → HBM4）
  ├─ 下游客户结构
  ├─ 热卖规格
  ├─ 用户/买家痛点
  ├─ Reddit 讨论热点
  ├─ YouTube 关键频道
  ├─ Twitter/X 关键账号
  ├─ 行业关键词
  └─ 商业模式
```

这份文件最值钱的信息不是产品列表——是**商业模式**那条：
> HBM 不是按传统内存逻辑卖的。它和 GPU 深度绑定，客户不是在"买内存条"，而是在锁定未来 GPU/ASIC 平台的合格供应。定价权不在内存厂那里——在封装良率那里。

这是一条普通的输出，但它是**可验证的**——可以溯源到公开的封装良率数据和 NVIDIA 供应链协议。和一篇散文里的"行业竞争激烈"有本质区别。

**Step 2: 竞品深潜**
```
输入："逐家分析 SK Hynix / Samsung / Micron"
输出：competitor-analysis.md
  └─ 三家公司官网讲的是完全不同的故事：
       • SK hynix → "AI Memory"（我不是内存厂，我是 AI 基础设施）
       • Micron → 供应链稳定性（美国本土安全，地缘政治牌）
       • Samsung → 技术代际，HBM4 base die 定制（暗示下一代反超）
```

不从官网话术里提取，你永远不会知道三家公司打的不是同一场战争。

**Step 3: 社媒分析**
```
输入："分析 SemiAnalysis / Asianometry / Dylan Patel"
输出：social-analysis.md
  └─ 社区最热的两个问题：
      1. "为什么 HBM 这么贵"
      2. "Samsung 什么时候能追上"
  └─ 这两个问题的答案 = 整个行业的核心矛盾
```

### 验证
这份数据库的价值不靠"感觉靠谱"，靠的是：
- 每一条关键信息可溯源（官网 / 财报 / 访谈）
- 核心矛盾可交叉验证（Reddit 热点 ↔ 商业模式 ↔ 竞争格局）
- 预测可事后检验（良率谁先跑稳 → 定价权归属）

### 可迁移性
这套管线对任何"听说很重要但摸不透"的领域都跑得通：
- 光模块 → 供应商（中际旭创 / Coherent）→ 下游（英伟达 / 谷歌）→ 风险（关税 / 产能）
- 液冷 → 供应商（CoolIT / Boyd）→ 技术路线（冷板 vs 浸没）→ 瓶颈（TCO / 标准）
- CoWoS 封装 → 台积电产能 → 下游分配 → 替代方案

**换一个行业名丢进去，管线跑出来的结构密度是一样的。**

## 核心原则

| # | 原则 | 说人话 |
|---|------|--------|
| 1 | **结构先于内容** | 先定义输出模板（字段/维度），再往里填信息。不是写作文，是填表 |
| 2 | **可溯源性** | 每一条关键输出附来源 URL。不可溯源的标注 ⚠️ |
| 3 | **聚焦矛盾** | 最有价值的信息不是"市场规模 100 亿"——是"定价权在封装良率" |
| 4 | **一步一停** | 每步完成暂停，问用户："接下来深挖哪个方向，还是够了？" |
| 5 | **时间戳硬化** | 行业地图逐月贬值。输出必须标注生成时间，后续引用时检查老化 |

## 常见误区

- ❌ 把 domain-mapper 当成 product-spec-builder 的前置步骤（它们是独立的，只是可以配合）
- ❌ 一把梭跑完 5 步不问用户（每步后必须暂停让用户决定方向）
- ❌ 输出"漂亮但空洞"的散文（每条信息必须有骨架：什么 / 谁 / 为什么 / 来源）
- ❌ 忽略"意外发现"（跑完 Step 2 后让 AI 标出最反直觉的 3 个点，那往往是核心入口）
