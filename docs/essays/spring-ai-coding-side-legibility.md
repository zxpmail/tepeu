# 代码：人类可读 vs Agent 可读

> **我让 Claude Code 在 Spring AI 业务代码库里改代码，对照两种写法：传统 Spring AI（分层、为人类设计）vs 扁平化（集中、为 Agent 设计——注册集中 + 路径注入）。看 Agent 跑多少轮。**
>
> **结果：** flash 上扁平 ≈ 传统；glm 上扁平少跑 25%；调试 / 重构任务两者各有胜负，不是一边倒。
>
> **怎么办：** (1) 注册集中化必做；(2) 路径看你模型默认读哪里（DeepSeek 吃 system，glm 吃代码库 md）；(3) 调试 / 重构看任务，别预设哪个好。
>
> **为什么：** Agent 不读你写的文档，它读自己默认会读的位置——放错等于没放。
>
> **边界：** 扁平化不是银弹——看需求。1 个代码库、不是定律、不是客服准确率。PithTrain 从零造能 -62%，本文只能拿回 flash 持平 / glm -25%。**数据：** `experiments/ate-bench/results/`。

* * *

## 1. 我做了什么

让 Claude Code（同一 CLI、同一模型）在 Tepeu 代码库里改代码、加工具、定位 bug，看它跑多少轮（`num_turns`）。

**对照两种写法：**

- **传统（默认）**——Spring AI 出厂形态。DI 找 Bean、注解扫描、`@Tool` / `ToolCallback` / `Advisor` 框架回调、AOP 切面。这套抽象为人类模式识别而设计：人靠 IDE 跳转和文档理解，Agent 要逐字追踪。加一个新工具，Agent 要串 `ChatController` → `AgentOrchestrator` → `ChatService`+`ToolCallbacks` → 各 `XxxTools` 类 → `ToolCallingAdvisor`——**5 个文件，静态阅读一次闭合不了**。

- **扁平化（改造后）**——在传统基础上叠加两件改造：
  1. **工具注册集中化**：从"散落在 `ChatService` 多处"改成"`Tools.java` 一行 `register` + `ToolCallbacks.from(toolRegistry.beans())`"
  2. **调用路径注入**：把上面那条 5 文件链路写进 Agent 默认会读的位置——
     - **A-only**：system 提示词
     - **B**：代码库可检索的 md 文档
     - **A+B**：两者都做

幂等（`IdempotencyService`）和记忆（`MemoryFileMirror`）留在基础设施——这是工程实现，**不在本文测度内**。

* * *

## 2. 结果

| 模型 | 任务 | 传统 | 扁平化 | 差 |
|---|---|---|---|---|
| flash | 理解调用链 | ~21 轮 | 11.6 轮（注入） | **-45%** |
| flash | 扩展·检索型 | 11 | 7（仅注册） | **-36%** |
| flash | 扩展·生成型 | 17 | 9.67（A-only）/ 12（A+B） | **-43%** / -29% |
| glm | 扩展·生成型 | 16 | 17（A-only） | +6%（变差） |
| flash | 调试 / 重构 | — | 更差 / 持平 | 更差 / 持平 |
| **flash 合计** | 扩展任务 | 25.7 | 24.7（A+B ×3） | **-4%** |
| **glm 合计** | 扩展任务 | 38 | 28.7（A+B ×3） | **-25%** |

**读这张表：**

- **flash 合计几乎打平（-4%）**——扩展·检索型变差抵消了扩展·生成型改善
- **glm 合计明显少（-25%）**——检索 / 生成同向改善
- **A-only 对 glm 反而变差**——glm 不读 system
- **调试 / 重构看任务**——T6/T8 上扁平化更差或持平，不是一边倒

注：不同实验日 Baseline 略有差异（T3 在 11~17 之间），原始记录见附录。

* * *

## 3. 为什么

Agent 默认读哪里，**因模型而异**：

- **DeepSeek 系**（flash / pro）默认吃 **system 上下文**——A-only 在扩展·生成型上下降明显
- **glm-5.2** 默认吃 **代码库目录结构**——A-only 在扩展·生成型上反而变差，A+B（含代码库 md）才有效

这就是说：**Agent 不读你写的文档，它读自己默认会读的位置。** 把调用路径写进 system 对 DeepSeek 有效、对 glm 无效；写进代码库 md 对 glm 有效、对 DeepSeek 中性。**放错位置等于没放。**

注：glm 上 A+B 的 -25% 尚未拆解（没跑 glm × 仅注册条件），所以这条机制是**工作假设**，不是结论。

* * *

## 4. 怎么办

三条结论：

1. **工具注册集中化必做**——跨模型、跨任务都有效，没坏处。
2. **调用路径放哪里，看你用的模型默认读哪里**——DeepSeek 系吃 system，glm 吃代码库 md。**别赌单一通道**，对你的模型冒烟三种放置方式（system / md / 都放），选赢家。
3. **调试 / 重构任务看具体情况**——A+B 在 T6/T8 上更差或持平，不预设哪个好。这类任务直接让 Agent 翻代码就行。

**改动频率参考：**

- **每周多次改工具 / 调用链** → 选定一个信息通道（system 或 md，看模型）进 CI；改前用 `generate_call_path_inject.py` 生成注入、`check_inject_sync --mode fail` 防漂移。
- **每季或更少** → 不必动结构，改前临时把路径贴进 prompt 即可。

* * *

## 5. 边界

- **样本规模**：n=3 / n=10；主锚 flash，pro / glm 为对照。
- **不能从"编码 turns ↓"推"客服准确率 ↑"**——编码 Agent 不接触业务 KPI；turns 只反映"代码库好不好改"。
- **标题是性质，实验是行为**——"可读"描述代码风格，"轮数"测操作成本；两者之间有跳跃。本文只回答"哪些任务上扁平化让 Agent 少跑"，不回答"哪种风格更可读"。
- **历史数字 -35% / -24% 已过时**（单场、未控制变量），**勿引**。
- **glm A+B ×3 的 Baseline 曾 API 529**，已重试补齐。

（"1 个代码库 / 不是定律" 和 "PithTrain -62% 不可类比"已在顶部摘要声明，不重复。）
