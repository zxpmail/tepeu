# 改 Spring AI 代码：两项改造能让 Agent 少跑几轮？

> **我做了什么：** 用 Claude Code 在 Spring AI 业务库（Tepeu）里改代码，对照「传统写法」与「两项改造」（工具注册集中 + 调用路径注入），看 Agent 的 `num_turns`。
>
> **结果（扩展任务合计）：** flash 约 **-4%**（几乎打平）；glm 约 **-25%**。单任务有涨有跌；调试 / 重构上改造后轮数往往更多。
>
> **可执行结论：** (1) 注册集中值得做；(2) 路径注入要放在**该模型实际会读的位置**，先对你的模型冒烟再定；(3) 调试 / 重构别预设改造更好。
>
> **边界：** 单仓、小样本、测的是「改代码轮数」不是「谁更可读」或「客服准确率」。与 PithTrain 从零造（约 -62%）不可直接类比。数据：`experiments/ate-bench/results/`。

* * *

## 1. 我做了什么

固定同一 CLI（Claude Code）、同一任务 JSON，在 Tepeu 上改代码 / 加工具 / 定位问题，主指标是 `num_turns`。

**对照：** 传统 Spring AI 分层写法 vs **两项改造**（不是推倒重写）。

| | 传统（baseline） | 改造后 |
|---|---|---|
| 改造 1·注册 | 工具在 `ChatService` 等多处装配 | **注册集中**：`Tools.java` + `ToolRegistry` → `ToolCallbacks.from(...)` |
| 改造 2·路径 | Agent 自己翻调用链 | **路径注入**：把 `ChatController → … → ToolCallingAdvisor` 写入指定通道 |

路径注入的**通道**（内容相同，放置不同）：

| 代号 | 放置 |
|---|---|
| A-only | system 提示词 |
| B | 仓库内可检索 md |
| A+B | 两者都放 |
| 仅注册（C） | 只做改造 1，无路径注入 |

幂等、记忆双写等是基础设施，**不在本实验测度内**。

* * *

## 2. 结果

先看**扩展任务合计**（主结论），再看分项（避免被单行大百分比带偏）。

### 2.1 合计（扩展任务）

| 模型 | 传统合计 | 改造后合计 | 相对变化 | 条件 |
|---|---|---|---|---|
| flash | 25.7 | 24.7 | **≈ -4%**（打平） | A+B ×3 |
| glm | 38 | 28.7 | **≈ -25%** | A+B ×3 |

flash 上分项有赢有输，合计被抵消；glm 上合计改善更明显。

### 2.2 分项（按条件分行；勿把不同条件的百分比横比成「全面更好」）

| 模型 | 任务 | 条件 | 传统 | 改造后 | 差 |
|---|---|---|---|---|---|
| flash | 理解调用链 | 路径注入（n≈10） | ~21 | 11.6 | -45% |
| flash | 扩展·检索型 | 仅注册（C） | 11 | 7 | -36% |
| flash | 扩展·生成型 | A-only | 17 | 9.67 | -43% |
| flash | 扩展·生成型 | A+B | 17 | 12 | -29% |
| glm | 扩展·生成型 | A-only | 16 | 17 | **+6%**（变差） |
| flash | 调试 T6 | A+B 类 ×3 | 6 | 9.67 | **+61%**（更差） |
| flash | 重构 T8 | A+B 类 ×3 | 5 | 6.67 | **+33%**（更差） |
| flash | T7 | A+B 类 ×3 | 15 | 15.67 | +4%（近乎打平） |

§2.1 的合计只对应 **A+B ×3 扩展任务**；上表其它行是辅证。不同实验日 Baseline 略有波动（如 T3 约 11–17）；原始记录见附录。

* * *

## 3. 为什么（工作假设，未完全拆开）

观察到的差异，更像「**注入放在哪**」与「**模型默认读哪**」是否对齐，而不是「改造标签」本身魔法：

| 放置 | 在本实验中的现象（有限样本） |
|---|---|
| A-only（system） | flash 扩展·生成型明显降轮；**glm 同任务变差** |
| A+B（含仓库 md） | glm 扩展合计约 -25% |

因此：文档写在仓库里，不等于会被读到；有效的是模型实际会碰的那条通道。

**尚未闭合的因果：** glm 的 A+B -25% **没有**拆成「仅注册 vs 注册+md」。把改善归因于「glm 更吃仓库 md」仍是**工作假设**。换模型 / 换仓前应自己冒烟，不要当定律抄。

* * *

## 4. 怎么办

1. **工具注册集中**——本实验里跨任务未见明显伤害，收益稳定，建议做。  
2. **路径注入先冒烟再固化**——对你常用的模型试 system / md / 都放，看哪条降轮；**不要默认「DeepSeek=system、glm=md」为行业规律**（本文样本太小）。  
3. **调试 / 重构**——T6（6→9.67）、T8（5→6.67）改造后更费轮；T7 近乎打平。这类任务直接让 Agent 读代码即可，不必先上全套注入。

**改动频率：**

- **经常改工具 / 调用链** → 选定赢家通道进流程；可用 `generate_call_path_inject.py` 生成、`check_inject_sync --mode fail` 防漂移。  
- **很少改** → 不必改仓库结构，临阵把路径贴进 prompt 即可。

* * *

## 5. 边界

- **样本**：主锚 flash；glm 等为对照；重复多为 n=3，理解类注入约 n=10。  
- **测的是什么**：编码 Agent 改库的轮数 / 成本，**不是**业务 Agent 准确率。  
- **测不到的东西**：本文不回答「哪种风格对人类更好读」，只回答轮数。  
- **过时数字**：历史单场 -35% / -24% **勿引**（未控制变量）。  
- **与 PithTrain**：对方从零造框架；本文是在已有 Spring AI 厚栈上叠两项改造，量级不可横比。

**复现：** `experiments/ate-bench/README.md`（扩面 / 第二模型 / Docker 实测已裁切，见 ADR-009）。  
**相关论文：** [Lai et al., arXiv:2605.31463](https://arxiv.org/abs/2605.31463)。
